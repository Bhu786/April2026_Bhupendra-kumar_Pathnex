# 🚀 Kubernetes (K8s) – Interview Ready Notes

## 🔹 What is Kubernetes?

Kubernetes is an **open-source container orchestration tool** used to:

* Manage containers
* Scale applications
* Ensure high availability
* Handle **auto-healing & disaster recovery (DR)**

---

## 🔹 Cluster

A **Cluster** is a collection of machines (nodes) working together.

* 1️⃣ Control Plane (Master)
* 2️⃣ Worker Nodes

---

# 🧠 Control Plane Components

### 1. API Server

* Entry point of Kubernetes
* All communication goes through this
* Acts like a **hub**

---

### 2. etcd

* Database of Kubernetes
* Stores all cluster state in **key-value format**
* Example: Pods, nodes, configs

👉 💡 "etcd = Brain + Memory"

---

### 3. Scheduler

* Assigns Pods to Nodes
* Decides **where container should run**

---

### 4. Controller Manager

* Maintains desired state
* Creates/deletes resources automatically

---

### 5. Cloud Controller Manager (CCM)

* Integrates with cloud providers like:

  * AWS (ELB, EBS)
  * Azure
  * GCP

---

# ⚙️ Worker Node Components

### 1. Kubelet

* Talks to API Server
* Ensures containers are running as expected

---

### 2. Kube-Proxy

* Handles networking
* Maintains routing rules

---

### 3. Container Runtime (CRI)

* Runs containers (Docker / containerd)
* Pulls images from registry

---

# 🔄 Controllers (Very Important)

| Controller      | Work                     |
| --------------- | ------------------------ |
| ReplicaSet      | Maintains number of Pods |
| Deployment      | Manages ReplicaSets      |
| Job             | One-time task            |
| CronJob         | Scheduled job            |
| Node Controller | Monitors nodes           |
| StatefulSet     | Stateful apps            |
| DaemonSet       | One Pod per node         |

---

# 🔁 Kubernetes Flow (MOST IMPORTANT)

👉 YAML → Desired State

```
User → YAML file apply
      ↓
API Server
      ↓
etcd (store state)
      ↓
Controller Manager
      ↓
Compare:
Actual State vs Desired State
      ↓
Fix mismatch (create/delete/update)
```

---

# 📦 Pod & ReplicaSet

### 🔹 Pod

* Smallest unit in Kubernetes
* Contains 1 or more containers

👉 Real world: **1–3 containers typical (not 6 usually)**

---

### 🔹 ReplicaSet

* Ensures fixed number of Pods always running

Example:

* 3 Pods needed
* 1 dies → new one created

---

# 🚀 Deployment

* Manages Pods using ReplicaSet
* Supports:

  * Rolling updates
  * Rollbacks

---

# 🌐 Services (Networking)

A **Service** gives:

* Stable IP
* DNS name
* Load balancing

---

### Types of Services

| Type         | Use                      |
| ------------ | ------------------------ |
| ClusterIP    | Internal communication   |
| NodePort     | External via Node IP     |
| LoadBalancer | Cloud load balancer      |
| Headless     | Direct Pod communication |
| ExternalName | Map external service     |

---

# 🔗 Pod Communication

* Pod → Pod (same cluster):
  ✔️ ClusterIP (normal)
  ✔️ Headless (direct DNS)

---

# 🧱 Namespace

* Logical separation inside cluster

### Default Namespaces:

* default
* kube-system
* kube-public
* kube-node-lease

👉 ❗ Correction:
Namespaces **do share cluster resources**, but logically isolated hote hain

---

# 💾 StatefulSet

Used for **stateful applications**

### Features:

* Stable name (pod-0, pod-1)
* Stable storage (PVC)
* Stable network identity

👉 Example:

* Databases (MySQL, MongoDB)

---

# 🔥 DaemonSet

* Runs **1 Pod on each node**

👉 Use cases:

* Logging agents
* Monitoring agents

---

# ⏰ Jobs & CronJobs

* Job → Runs once
* CronJob → Runs on schedule

---

# 🌍 Ingress (VERY IMPORTANT)

Ingress = **HTTP/HTTPS routing layer**

* Exposes services using:

  * Domain name
  * Path-based routing

---

## 🔁 Ingress Flow

```
User (browser)
   ↓
Ingress Controller (NGINX / API Gateway)
   ↓
Service
   ↓
Pod
```

👉 ❗ Correction:
NGINX Ingress **closed nahi hai**

# ⚠️ Interview Questions (Corrected)

---

### ❓1. If API Server goes down?

✔️ Already running apps → **continue running**
❌ New changes → **NOT possible**

👉 No scaling / deployment until API server is back

---

### ❓2. Why 3 nodes (etcd quorum)?

* Uses **quorum-based decision**
* Odd number needed

Example:

* 3 nodes → 2 majority → decision possible

---

### ❓3. Can worker node run on master?

✔️ Yes (in small setups)
❌ Not recommended in production

---

# 💡 Important Points

* Pods are **ephemeral**
* Use **Deployment for stateless**
* Use **StatefulSet for stateful**
* Services provide **stable access**
* Controllers maintain **desired state**
